## 3. 브랜드 해석

## 3.1 Sponzey 브랜드 톤

Sponzey 사이트는 기술 제품임에도 차갑고 기업적인 SaaS 톤보다 다음 쪽에 가깝다.

- 빠르게 이해되는 구조
- 밝고 신뢰감 있는 첫인상
- 과도한 시각 효과보다 또렷한 정보 구획
- 검은 외곽선과 큰 라운드로 만드는 친근한 도구감
- 설명, 안내, 다운로드, 활용법을 바로 연결하는 실무 중심 카피

즉, Sponzey FileSharing도 "복잡한 엔터프라이즈 네트워크 툴"처럼 보여서는 안 된다.  
"내부망에서 바로 쓰는 빠르고 믿을 수 있는 실무용 데스크톱 도구"처럼 보여야 한다.

## 3.2 FileSharing 제품으로 옮길 때의 해석

SponzeyBoard가 "화면 위에서 바로 설명하는 빠른 도구"라면 Sponzey FileSharing은 "로컬 네트워크에서 바로 보내고 바로 확인하는 빠른 도구"다.

따라서 FileSharing의 디자인 핵심 키워드는 아래 5개로 정의한다.

- Fast
- Visible
- Trusted
- Local
- Practical

## 3.3 디자인 컨셉 이름

### 컨셉명: `Warm Utility Courier`

설명:

- `Warm`: Sponzey의 옐로우 기반 친근한 인상
- `Utility`: 장식보다 실무 도구로서의 기능 우선
- `Courier`: 노드 간 파일을 전달하는 제품의 역할을 직관적으로 표현

이 컨셉은 "귀여운 파일 앱"이 아니라 "따뜻하지만 단호하고 빠른 전송 도구"를 의미한다.

## 4. 제품 시각 전략

## 4.1 전체 방향

앱은 전체적으로 밝은 배경 위에 굵은 외곽선 카드가 얹힌 구조를 기본으로 한다.

중요한 원칙:

- 배경은 가볍고 밝게
- 정보 블록은 카드 단위로 또렷하게
- 주요 액션은 브랜드 옐로우로 강조
- 전송 상태와 네트워크 상태는 색보다 구조와 아이콘으로 먼저 구분
- 경고/실패만 강한 상태 컬러를 사용

## 4.2 왜 이런 방향이 맞는가

plan 문서의 핵심 요구는 아래와 같다.

- 피어 탐색
- 인증 상태 확인
- 다중 세션 진행률 모니터링
- 전송 성공/실패 추적
- 로그와 이력 조회

이런 제품은 정보량이 많다.  
따라서 과한 그래픽이나 유리질감보다, `명확한 프레임`, `읽기 쉬운 목록`, `빠른 상태 파악`이 우선이다.

Sponzey 사이트의 검은 외곽선 카드와 큰 라운드는 이 목표와 잘 맞는다.

## 5. UX 원칙

### 원칙 1. 상태는 항상 눈에 보여야 한다

사용자는 항상 아래를 바로 확인할 수 있어야 한다.

- 지금 로그인된 사용자
- 현재 온라인 피어 수
- 현재 진행 중 전송 수
- 가장 위험한 상태 하나
- 마지막 오류 또는 인증 실패

### 원칙 2. 전송은 "작업"과 "세션"이 분리되어 보여야 한다

사용자는 "파일 1개를 5명에게 보냈다"를 하나의 작업으로 인식한다.  
하지만 내부적으로는 여러 세션이다.

UI는 이 둘을 함께 보여야 한다.

- 상위: 배치 작업
- 하위: 대상별 세션

### 원칙 3. 빠른 시작이 가장 중요하다

앱을 열고 5초 안에 다음 중 하나를 할 수 있어야 한다.

- 온라인 피어 확인
- 파일 드래그 앤 드롭
- 최근 전송 재확인

### 원칙 4. 경고는 강하게, 평상시는 차분하게

항상 노란색이 지배하는 화면이 아니라, 평상시에는 밝은 종이 같은 화면을 유지하고 위험 상태에서만 빨강/주황을 쓴다.

### 원칙 5. 네트워크 도구처럼 보이되 네트워크 장비 UI처럼 보이지 말 것

전문 네트워크 콘솔처럼 과도하게 기술적이면 일반 사용성이 떨어진다.  
반대로 소비자용 클라우드 드라이브처럼 가볍게 보이면 인증/보안/신뢰감이 약해진다.

중간 지점이 필요하다.

### 원칙 6. 모든 핵심 액션은 1단계 깊이 안에 있어야 한다

- 파일 보내기
- 피어 선택
- 수신 승인
- 실패 재시도
- 저장 폴더 열기

위 동작은 서브메뉴 두세 단계를 파고들지 않게 해야 한다.

## 7.3 피어 화면

### 목적

- 자동 탐색된 피어 확인
- 허용/차단/즐겨찾기 관리
- 연결과 전송의 출발점 역할

### 화면 구조

- 상단: 페이지 타이틀 + 검색 + 필터 + `새로 탐색`
- 본문: 카드형 그리드 또는 리스트
- 우측 드로어: 선택된 피어 상세

### 피어 카드 요소

- 장치 아이콘
- 표시 이름
- `user_id`
- 장치 이름
- OS 뱃지
- 온라인 상태 점
- 마지막 응답 시간
- 프로토콜 버전
- 버튼: `연결`, `파일 보내기`

### 상태 뱃지 규칙

- Online: 초록 점 + `온라인`
- Idle: 회색 점 + `대기`
- Busy: 노란 점 + `전송 중`
- Incompatible: 빨간 테두리 + `버전 다름`
- Blocked: 회색 outline + `차단됨`

### 상세 드로어

- 신뢰 상태
- 허용 여부
- 최근 전송 기록
- 인증 가능 여부
- 지원 기능
- 빠른 액션

## 7.4 인증 요청 / 연결 승인 UI

### 목적

- password-derived JWT 상호 인증 흐름을 사용자에게 설명 가능하게 표현

### 요구사항

- 기술 내부 구현은 복잡하지만 UI는 복잡하게 보이지 않아야 한다

### 상태 단계 표기

1. 연결 요청 보냄
2. 상대 장치 확인 중
3. 인증 토큰 검증 중
4. 세션 수립 중
5. 연결 완료

### 시각 표현

- 모달 또는 플로팅 패널
- 단계별 스텝 인디케이터
- 실패 시 어디서 실패했는지 한 줄 설명

예:

- "허용되지 않은 사용자입니다."
- "인증 토큰이 만료되었습니다."
- "상대 장치와 시간이 맞지 않습니다."

### 디자인 원칙

- JWT, nonce, jti 같은 용어는 기본 화면에 직접 드러내지 않는다
- 고급 로그 보기에서만 기술 세부를 노출한다

## 7.5 전송 화면

### 목적

- 파일 전송 생성과 진행 모니터링

### 화면 구조

- 상단 액션 바
- 좌측: 파일 드롭존 / 배치 구성
- 우측: 대상 선택 패널
- 하단: 전송 시작 버튼

### 파일 드롭존

- 넓은 카드형 영역
- 배경은 거의 흰색
- 내부에 얇은 점선이 아니라 검은 외곽선 기반 큰 박스
- 텍스트: "파일을 여기로 끌어오거나 선택하세요"

### 대상 선택 패널

- 최근 대상
- 온라인 피어 목록
- 다중 선택 체크박스
- 선택 수 표시

### 전송 옵션 영역

- 저장 정책
- 자동 승인 여부 안내
- 동시 전송 수
- 파일별 예상 크기

### 하단 CTA

- 메인 버튼: `전송 시작`
- 보조 버튼: `초기화`

## 7.6 전송 큐 / 진행 모니터

### 목적

- 전송 상태를 배치와 세션 단위로 추적

### 시각 구조

- 아코디언 리스트
- 상위 카드: 배치 전송
- 내부 하위 행: 대상별 세션

### 배치 카드 정보

- 배치 ID 또는 사용자 친화적 제목
- 파일 수
- 대상 수
- 총 용량
- 전체 진행률
- 상태 요약

### 세션 행 정보

- 수신자
- 현재 파일
- 속도
- 남은 시간
- 재전송 횟수
- 진행률 바
- 상태 뱃지

### 진행률 바 디자인

- 기본 트랙: 연한 아이보리
- 채움: 브랜드 옐로우
- 실패 chunk 존재 시 얇은 빨간 마커
- 재시도 중이면 우측 끝에 미세한 움직임 표시

## 7.7 수신함 / 이력 화면

## 14. 화면별 상세 상태 정의

## 14.1 Empty State

필요 화면:

- 피어 없음
- 진행 중 전송 없음
- 수신 이력 없음
- 로그 없음

표현 방식:

- 작은 아이콘
- 짧은 한 줄 설명
- 바로 할 수 있는 액션 1개

예:

- "아직 발견된 장치가 없습니다."
- `탐색 다시 시도`

## 14.2 Loading State

- 전체 화면 스켈레톤보다 카드 단위 스켈레톤 선호
- 리스트는 3~5개 가짜 행
- 진행률이 있는 경우 spinner보다 단계 텍스트 우선

## 14.3 Error State

- 인증 실패
- 포트 충돌
- 저장 공간 부족
- 방화벽 차단
- 프로토콜 버전 불일치

오류 카드는 다음을 포함:

- 명확한 제목
- 짧은 설명
- 재시도 또는 설정 이동 버튼
- 고급 정보 보기

## 14.4 Success State

- 지나치게 축하하지 않음
- "전송 완료" + 파일 수/대상 수/경로 열기

## 18. 최종 디자인 결론

Sponzey FileSharing의 디자인은 Sponzey 사이트의 시각 언어를 그대로 복사하는 것이 아니라, 그 브랜드 문법을 데스크톱 네트워크 도구에 맞게 번역해야 한다.

그 번역 규칙은 아래 한 문장으로 정리할 수 있다.

### "밝고 친근하지만, 상태와 신뢰가 먼저 보이는 로컬 전송 도구"

실제 디자인 실행 시 가장 중요한 세 가지는 다음이다.

1. `#FEEA29 + Ink + 큰 라운드 + 2px 보더`라는 브랜드 문법을 유지할 것
2. 배치/세션/피어/이력을 명확히 구획하는 정보 구조를 우선할 것
3. 전송 상태와 인증 상태가 모든 화면에서 즉시 보이게 만들 것

이 방향이면 SponzeyBoard와 같은 패밀리 안에 있으면서도, FileSharing만의 성격이 분명한 제품 디자인을 만들 수 있다.
